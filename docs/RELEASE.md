# 릴리스·운영

해외 생활비 비교 앱의 버전 전략·브랜치 전략·릴리스 절차·스토어 제출·개인정보·고객 지원·재해 복구·마케팅을 한 곳에 모은 단일 출처. M5(자체 검증 완료) → M6(스토어 출시) 단계에 진입할 때 본 문서가 체크리스트가 된다.

## 1. 버전 전략 (Semantic Versioning)

| 변경 종류 | 예시 | 버전 bump | 배포 수단 |
|---|---|---|---|
| 데이터·문구 패치, UI 마이크로 수정 | "갱신: 2026-Q3 도시 데이터" | v1.0.x → v1.0.(x+1) | **EAS Update** (OTA) |
| 새 화면·새 카테고리·동작 변경 | "rent 상세 화면 추가" | v1.0.x → v1.1.0 | **EAS Update** 가능하면 (네이티브 미변경 시) |
| 호환성 깨지는 변경, 네이티브 의존성 변경 | "스키마 v2 도입", "권한 추가" | v1.x.x → v2.0.0 | **새 바이너리** 빌드 + 스토어 심사 |

규칙:
- `app.json` 의 `version` (스토어용) + `runtimeVersion` (OTA 호환 키) 동시 관리
- `version` 은 사람용, `runtimeVersion` 은 OTA-binary 호환 식별
- 데이터 스키마 변경은 무조건 `runtimeVersion` 을 올린다 (구 바이너리에 신 데이터 보내지 않음)

## 2. 브랜치 전략

1인 운영 + 하네스 기반.

| 브랜치 | 용도 |
|---|---|
| `main` | 항상 배포 가능. 모든 step 의 기본 도착지. |
| `feat-<phase-name>` | 하네스가 phase 시작 시 자동 생성. 모든 step 커밋이 여기에. phase 끝나면 main 으로 squash 또는 merge. |
| `hotfix-<issue>` | 출시 후 긴급 패치 (필요 시) |

규칙:
- main 에 직접 커밋 금지 (하네스 자동 커밋도 feat 브랜치)
- phase 머지 후 feat 브랜치 삭제
- main 의 모든 head 는 적어도 lint + typecheck + test 통과
- 출시 직전 `release-v1.0.0` 같은 태그 생성

## 3. 환경·EAS 프로필

`eas.json` 의 3개 프로필:

| 프로필 | 용도 | 빌드 타입 | 환경변수 |
|---|---|---|---|
| `development` | 시뮬레이터·실기기 dev | dev client | `EXPO_PUBLIC_DATA_BASE_URL=https://raw.githubusercontent.com/<user>/<repo>/main/data/cities` (또는 dev URL) |
| `preview` | TestFlight·Internal Play 베타 | release | 동일 (또는 staging URL) |
| `production` | App Store·Play Store | release | production URL |

환경변수 명명 규약:
- `EXPO_PUBLIC_*`: 클라이언트 노출 OK (베이스 URL 등)
- 비밀값: v1.0 에 없음 (계정·서버 없으므로). 추후 도입 시 `expo-secure-store` + `eas env`.

## 4. 릴리스 절차 (M6)

M0~M5 는 본인 디바이스 + Expo Go 로 무료. M6 진입 시 비용·심사 발생.

```
M5 종료 (본인 디바이스 검증 완료)
  ↓
[1] 결제
  ├─ Apple Developer Program $99/년
  └─ Google Play Console $25 (1회)
  ↓
[2] 자산 확정
  ├─ assets/icon.png (1024×1024)
  ├─ assets/splash.png
  ├─ 스크린샷 (스토어 사양에 맞춰 6장 이상, 각 사이즈)
  └─ 앱 메타데이터 (§5)
  ↓
[3] 베타 빌드
  └─ eas build --profile preview --platform all
     → TestFlight (iOS) + Internal Play track (Android)
     → 지인 5~10명 베타 1주
  ↓
[4] 베타 피드백 반영
  └─ 크래시·UX 이슈 패치 (EAS Update 또는 새 빌드)
  ↓
[5] 프로덕션 빌드
  └─ eas build --profile production --platform all
  ↓
[6] 스토어 제출
  ├─ App Store Connect: 메타데이터 + Privacy Nutrition Label + 심사 제출
  └─ Play Console: 메타데이터 + Data Safety form + 출시 제출
  ↓
[7] 심사 (Apple ~1~3일, Google ~1~7일)
  ↓
[8] 승인 → 단계적 출시 (Play 는 5%→20%→100% rollout 권장)
  ↓
[9] 출시 후 모니터링 (수동)
```

## 7. 개인정보 처리방침 (한국어 초안)

스토어 제출용 정적 페이지. GitHub Pages 또는 별도 호스팅. URL 변경 가능성을 고려해 앱에서 항상 외부 링크로 노출.

```
# 해외 생활비 비교 앱 개인정보 처리방침

본 앱은 사용자 개인정보를 **수집하지 않습니다**.

## 수집·저장 정보
- 본 앱은 어떠한 사용자 정보도 외부 서버로 전송하거나 수집하지 않습니다.
- 사용자가 선택한 페르소나(유학생/취업자), 즐겨찾기 도시, 최근 본 도시는
  사용자 기기 내부 저장소(AsyncStorage)에만 저장됩니다.
- 앱 삭제 시 위 데이터는 모두 함께 삭제됩니다.

## 외부 서비스
- 환율 데이터: open.er-api.com 의 무료 환율 API 를 호출합니다.
  사용자 식별 정보는 전송되지 않으며, 일반적인 IP 기반 요청만 발생합니다.
- 도시 비용 데이터: GitHub 공개 저장소에서 정적 JSON 파일을 다운로드합니다.
  사용자 식별 정보는 전송되지 않습니다.

## 분석·추적
- 본 앱은 어떠한 분석 도구·광고 식별자·추적 SDK 도 사용하지 않습니다.

## 데이터 정확성 고지
- 본 앱이 표시하는 모든 비용 정보는 공공 통계·공식 기관 발표 등을 기반으로
  분기마다 수동 큐레이션됩니다.
- 실제 비용은 시점·지역·개인 상황에 따라 다를 수 있으며,
  본 앱의 정보를 의사결정의 단일 근거로 삼지 마십시오.

## 변경
- 본 처리방침은 변경 시 앱 설정 화면에서 안내됩니다.

## 문의
- <feedback 이메일>

마지막 갱신: <YYYY-MM-DD>
```

## 13. 모니터링·로깅

### 모니터링
- v1.0: 분석 도구 0건. 다운로드·평점은 스토어 콘솔 수동 확인 (주 1회).
- v1.1 검토: Sentry crash reporting 도입 가능성 (별도 ADR + 사용자 동의 추가).

### 로깅 정책
- 개발: `console.log`, `console.warn`, `console.error` 자유 사용.
- 프로덕션: ESLint `no-console` 규칙으로 `console.log` 차단. `console.warn/error` 만 허용.
- Babel 플러그인 (production 빌드): `transform-remove-console` 로 `log`/`debug` 제거 (warn/error 보존).
- 사용자에게 로그 노출 금지 (개발자 메뉴 등 v1.0 미도입).

## 13.5 사용자 버그·이슈 리포트 흐름

```
사용자 발견
   │
   ├─→ 앱 내 [피드백 보내기] 메뉴 (mailto:)
   │     │
   │     └─→ 이메일 자동 prefill: 디바이스·OS·앱 버전 + 사용자 작성
   │
   ├─→ 앱 스토어 평점·리뷰
   │     │
   │     └─→ 스토어 콘솔에서 운영자가 주 1회 모니터
   │
   └─→ (v1.1) 인앱 항목별 신고 🚩 (ADR-010)
         │
         └─→ 자동 prefill: 도시·항목·현재값·디바이스
   
운영자 처리
   │
   ├─→ 데이터 정정 요청: 다음 분기 자동 PR 또는 수동 PR
   ├─→ 앱 버그: GitHub issue 등록 → 우선순위 (P0~P3)
   │     │
   │     ├─→ P0 (크래시): 1주 이내 hotfix (§13.6)
   │     ├─→ P1 (기능 미작동): 다음 minor 릴리스
   │     └─→ P2~P3: backlog
   └─→ 회신 SLA: 1주 이내 best-effort
```

## 13.6 Hotfix 절차 (출시 후 긴급 패치)

P0 (크래시·데이터 손상·심각한 시각 깨짐) 발견 시:

```
1. issue 등록 (GitHub) — P0 라벨
2. 재현 테스트 작성 → 실패 확인 (TDD)
3. fix 구현 → 재현 테스트 통과
4. 분기점 확인:
   ├─ JS-only 변경 → EAS Update OTA (당일 배포 가능)
   └─ 네이티브 의존성 변경 → 새 바이너리 + 스토어 심사 (Apple ~24h expedited 가능, Google ~6h)
5. 빌드 & 배포
6. 검증 (영향받은 기기 5개 이상 즉시 확인)
7. 사용자 공지 (스토어 changelog + 앱 내 toast)
```

**Expedited review 활용:**
- Apple: App Store Connect → "Request Expedited Review" — 크래시·법적 이슈 시
- Google: 자체 review 6시간 내, 별도 요청 불필요

**Rollback:**
- EAS Update: 이전 OTA update channel 로 즉시 rollback (`eas update --channel production --message "rollback"`)
- 바이너리: 스토어에 잘못된 버전 노출 정지 + 이전 버전 재게시
- repo: `git revert <commit>` + 새 패치 빌드

## 14. 재해 복구·가용성

| 시나리오 | 대응 |
|---|---|
| GitHub raw URL 다운 | 시드 데이터로 fallback (ARCHITECTURE.md §캐시·오프라인). 사용자에게 "데이터 갱신 실패" 배지. |
| 환율 API 다운 | 마지막 성공 캐시 + 경고 배지. 24h 이상 stale 시 비교값 계산 보류. |
| 데이터 repo 삭제 | 미러 또는 jsDelivr CDN 백업 (출시 직전 결정). 백업 URL 을 baseURL 변경으로 즉시 전환 가능 (EAS Update 로 push). |
| 환율 API 정책 변경 | 다른 무료 API 또는 ECB 데이터로 전환. 변경 시 ADR. |
| 도메인·이메일 만료 | 1년 단위 갱신 알림. |
| Apple/Google 계정 정지 | 별도 대응 매뉴얼 v1.x 검토. |

## 15. 스키마 마이그레이션 정책

AsyncStorage 키에 **버전 suffix** 를 강제한다.

```
city:<id>:v1     # 도시 데이터 v1
fx:v1            # 환율 v1
persona:v1       # 페르소나 (Zustand persist 의 internal key)
```

스키마 변경 시:
1. 새 키 생성 (`city:<id>:v2`)
2. `runtimeVersion` 을 올려 OTA 분리
3. 신 코드는 v2 만 읽기. v1 캐시는 무시 + 자동 정리(`AsyncStorage.removeItem` 마이그레이션 함수)
4. 마이그레이션 함수 위치: `src/lib/migrations/v1ToV2.ts` 같은 파일
5. 마이그레이션 테스트는 TESTING.md 인벤토리에 추가

호환성 깨짐 결정은 ADR 기록.

## 16. 앱 업데이트 메커니즘

| 변경 | 수단 | 사용자 경험 |
|---|---|---|
| 데이터 갱신 (JSON repo) | repo commit | 24h 캐시 만료 후 자동 fetch (사용자 액션 불필요) |
| UI·문구 패치 | EAS Update (OTA) | 앱 다음 콜드스타트 시 자동 적용 (백그라운드 다운로드) |
| 새 기능 (네이티브 미변경) | EAS Update | 동일 |
| 네이티브 의존성 추가, 권한 변경 | 새 바이너리 + 스토어 심사 | 사용자가 스토어에서 업데이트 |

EAS Update 는 동일 `runtimeVersion` 안에서만 작동. 큰 변경은 새 바이너리 필수.

강제 업데이트 정책:
- v1.0: 강제 업데이트 미구현 (앱이 stale 코드여도 동작은 함)
- v1.x 검토: 데이터 스키마 변경 등 호환성 이슈 시 startup 체크 + 업데이트 안내 화면

## 17. 번들 크기·성능 측정

### 번들 크기
- 측정: `npx expo export --platform ios` → `dist/` 디렉터리 크기 + gzipped 추정
- 기준: 5 MB gzipped (ADR-017)
- 초과 시: 폰트 subset, 미사용 의존성 제거, 시드 데이터 별도 파일 분리 검토

### 부팅 시간
- 측정: 시뮬레이터 cold start (`Settings > General > Reset > Reset Content` 후 첫 실행)
- 기준: 3초 이내

### 측정 시점
- M5 step2 (a11y-perf) — 첫 측정
- 매 분기 데이터 갱신 시 + 매 minor 릴리스 전 — 회귀 검증
